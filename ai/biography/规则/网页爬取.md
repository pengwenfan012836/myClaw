# Claude Code 网页抓取方案深度对比

> **一句话总结**：5 种方案不是替代关系，而是不同技术层次的互补工具——从轻量 HTTP 请求到云端 SaaS，选对场景才能事半功倍。

---

## 技术层次概览

| 方案 | 层次 | 核心定位 |
|------|------|----------|
| WebFetch | 网络层 | 最轻量的 HTTP 请求工具 |
| Playwright MCP | 浏览器层 | 模拟真人操作的真实浏览器 |
| Scrapling | 爬虫框架层 | 自适应解析的 Python 爬虫框架 |
| Firecrawl | 云端服务层 | 网页数据提取的 SaaS 服务 |
| Agent-Reach | 聚合编排层 | 11+ 社交平台的统一接入层 |

> 层级递增，能力递增，复杂度也递增。互补关系，不是替代关系。

---

## 方案一：WebFetch —— 螺丝刀

Claude Code **内置工具**，零配置，开箱即用。

**工作方式**：发 HTTP 请求 → 拿 HTML → 转 Markdown 返回

### 优点

- 轻量，无需安装依赖
- 安全性高，内置沙箱
- 自动 HTTPS 升级
- 15 分钟缓存

### 缺点

- 不执行 JavaScript
- 不支持登录
- 不能点击、滚动、填表单
- 企业级反爬可能拦截

> WebFetch 拿到的是「服务器返回的原始文档」，**不是**「你在浏览器里看到的页面」

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 博客文章 | X/Twitter |
| 公开 API 文档 | SPA 单页应用 |
| 静态页面文本 | 需登录的页面 |

**实测**：抓 X 帖子，直接返回 `JavaScript is not available`

---

## 方案二：Playwright MCP —— 电钻

微软维护的浏览器自动化框架，通过 MCP 协议接入 Claude Code。

启动真实的 Chromium 浏览器，完整加载页面、执行 JS、渲染 DOM。

### 优点

- 拿到你眼睛看到的页面
- JS 渲染、动态加载全支持
- 能交互：点按钮、填表单、模拟滚动、截图
- 可以走完登录流程
- 微软维护，社区成熟
- 数据本地处理

### 缺点

- 较重，每次启动完整 Chromium
- 一次只能操作一个页面
- 不是为批量爬取设计

### 适用场景

| 适合 | 不适合 |
|------|--------|
| JS 渲染页面（SPA、React/Vue） | 大规模批量爬取 |
| 需登录的内容 | 高频率抓取 |
| 需交互操作的页面 | |
| 截图 | |

**实测**：X 帖子完整拿到全文、作者信息、互动数据（回复/转发/点赞/书签/浏览量）、图片链接

## 方案三：Scrapling —— 专业工具箱

Python 爬虫框架，GitHub 20.6K stars，92% 测试覆盖率。

### 三种 Fetcher 模式

| 模式 | 说明 |
|------|------|
| Fetcher | 普通 HTTP 请求（类似 WebFetch） |
| StealthyFetcher | 反反爬，内置绕过 Cloudflare |
| DynamicFetcher | 浏览器渲染（类似 Playwright） |

### 核心特性：自适应解析

网站改版后选择器失效？Scrapling 用相似度算法自动重新定位元素，DOM 结构变了也能找到对应内容。

### 优点

- 三种模式灵活切换
- 自适应解析，抗改版
- 支持异步并发、断点续爬、流式处理
- 内置 MCP Server
- 纯本地运行
- 解析速度比 BeautifulSoup 快 **400-600 倍**

### 缺点

- 需要 Python 3.10+ 环境
- 学习曲线陡峭
- 不适合交互场景

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 长期、批量数据采集 | 偶尔抓一个页面 |
| 需要应对反爬 | 杀鸡用牛刀 |

## 方案四：Firecrawl —— 装修公司

YC 孵化，$14.5M A 轮，GitHub 69K+ stars，35 万+开发者在用。

**云端 SaaS**，给一个 URL，在云端完成所有脏活：浏览器渲染、反爬绕过、内容清洗、结构化提取，直接返回干净的 Markdown 或 JSON。

### 支持功能

- 全站爬取
- 站点地图发现
- 搜索+抓取一体化
- `/agent` 智能导航

### 优点

- 省心，无需本地依赖
- 无需管理版本兼容
- 官方 MCP Server，一行命令接入
- 功能全面

### 缺点

- 隐私：内容经过 Firecrawl 服务器
- 费用：大量使用需付费
- 服务商依赖

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 大规模全站爬取 | 敏感数据 |
| 不想维护工具链 | 在意隐私的场景 |
| 抓公开信息 | |

## 方案五：Agent-Reach —— 万能转接头

不是独立爬虫引擎，而是**脚手架**——把 11+ 平台整合为统一入口。

### 底层工具

| 平台 | 工具 |
|------|------|
| YouTube | yt-dlp（148K stars） |
| X/Twitter | xreach CLI（cookie 认证） |
| 通用网页 | Jina Reader |

### 优点

- 本地运行，免费开源
- 多平台统一入口
- 省事

### 缺点

- 依赖链长
- 项目较新，稳定性待验证
- 维护成本高

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 多平台信息采集 | 单一平台深度抓取 |
| | 稳定性要求高的场景 |

## 选型建议

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 快速查看静态页面 | WebFetch | 零配置最轻量 |
| 需要 JS 渲染或登录 | Playwright MCP | 需要真实浏览器 |
| 需要点击、填表等交互 | Playwright MCP | 唯一能做到的方案 |
| 批量提取结构化数据 | Scrapling | 主战场 |
| 网站有 Cloudflare 防护 | Scrapling StealthyFetcher | 专门解决此问题 |
| 长期监控网站变化 | Scrapling | 自适应解析 + 断点续爬 |
| 大规模全站爬取 | Firecrawl | 云端处理，省心 |
| 跨多个社交平台采集 | Agent-Reach | 11+ 平台统一入口 |
| 在意数据隐私 | Scrapling / Agent-Reach | 纯本地运行 |
| 不想维护工具链 | Firecrawl | SaaS 零维护 |

## 推荐组合方案

### 日常组合：WebFetch + Playwright MCP

- 大部分场景用 WebFetch 先试
- 静态页面直接搞定
- 遇到 JS 渲染或需要交互的再切 Playwright
- 覆盖 **80%** 的日常需求

### 进阶组合：加上 Scrapling MCP

- 有批量采集或长期监控需求时加入
- 批量调度和自适应解析是前两者做不到的
- 三者都本地运行，数据不外传

### 全覆盖组合：再加 Agent-Reach

- 同时从 YouTube、B站、小红书等平台采集时加入
- 依赖链长，维护成本也高

---

## 几个容易忽略的点

### WebFetch 不等于浏览器

大部分网站都有 JS 渲染内容，WebFetch 适用范围比想象的窄。

### Playwright 不适合批量

启动完整 Chromium，一次操作一个页面，性能浪费严重。

### Firecrawl 的隐私问题

所有数据经过第三方服务器处理，商业敏感信息请选本地方案。

### Agent-Reach 本质是编排层

底层全靠第三方工具，升级依赖可能引发连锁问题。

---

## 总结

没有「最好」的方案，只有「最合适」的方案。

**建议路径**：WebFetch → 不够用了上 Playwright → 有批量需求再上 Scrapling

按需升级，别过度配置。
